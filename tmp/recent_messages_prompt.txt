Chat-scoped, not agent-scoped ``` tg:-1003894351534:topic:2166 ← isolated tg:-1003894351534:topic:1712 ← isolated tg:-1003894351534:dm:123456 ← isolated ``` The shared archive helps (`shared:architect:main:session:*`) but it's still chat-keyed under the hood. If the summary missed something — it's gone forever. Compare to MemPalace's verbatim approach: never delete, always search original text. ### 5. Only one active summary Each summarization run produces a new row. The prompt only uses the **latest** one. So you get: ``` Summary 1: messages 1-80 Summary 2: messages 81-200 ← only this one is used ``` There's no recursive summary-of-summaries. As chats get long, old context just disappears behind the latest summary window. ### 6. No time-range queries beyond 3 hardcoded windows ```python "today" → works "yesterday" → works "this week" → works "last month" → ❌ nope "between March 1 and 15" → ❌ nope ``` ### 7. No fact decay A fact from 6 months ago with confidence 0.78 stays "active" forever unless manually forgotten. No recency weighting, no decay curve, no automatic archival of stale facts. --- ## Bottom line It's a **solid SQLite-backed summary system** that does exactly what it was designed to do: keep recent context + a compressed summary + a handful of explicit facts. For the scale and scope of the existing bridge, it works fine. But compared to what MemPalace offers — semantic search, verbatim storage, agent diaries, knowledge graphs — this is a fundamentally different (and older) approach. The two gaps that hurt most: 1. **No semantic retrieval** — you can't search your own memory 2. **No automatic fact extraction** — the AI sees everything but remembers almost nothing unless explicitly told The bones are good. The gap is the retrieval and extraction intelligence on top.
